Kein roter Faden

Man sitzt im leeren Repo und fängt irgendwo an. Was kommt zuerst? Wo steht der Stand? Jeder macht es ein bisschen anders.

Die Idee
Skills sind geführte Befehle. Das Wissen steckt im Prozess.

Jeder Skill ist ein kleiner, geführter Workflow in Claude Code. Du tippst einen Befehl wie /init oder /frontend — und wirst durch den Schritt geleitet, mit allem p2cx-Wissen schon eingebaut. Kein leeres Blatt, keine vergessenen Standards.

  • Eine Frage nach der anderen — immer mit empfohlener Antwort. Du bestätigst oder korrigierst.
  • Der Stand lebt in GitHub-Issues, nicht im Chat. Nichts geht verloren.
  • Jeder Skill sagt, was als Nächstes kommt — du behältst die Kontrolle.
So fühlt es sich an

„Soll der Login über Azure AD laufen? Empfehlung: Ja — das ist p2cx-Standard."

„Braucht die App KI-Funktionen? Wenn ja, läuft alles über Langdock EU — DSGVO-konform, kein US-Provider."

„Ich habe 5 Features identifiziert. Hier die Reihenfolge — passt das so?"

Der Workflow — von der Idee bis Go-Live

Neun Skills, ein roter Faden. Jeder Schritt schreibt seinen Stand ins GitHub-Issue und übergibt auf Zuruf an den nächsten.

01 · Setup
/init

Legt das private Repo an, grillt dich zur Idee, schreibt das PRD und die Features als GitHub-Issues (P0/P1/P2).

02 · Spec
/write-spec

Spezifiziert ein Feature tief aus — User Stories, Edge Cases, Akzeptanzkriterien auf Deutsch. Direkt ins Issue.

03 · Design
/architecture

Technisches Design ohne Code: Komponenten, Datenmodell, Entscheidungen — verständlich für den Produkt-Kopf.

04 · Bau UI
/frontend

Baut die Oberfläche im p2cx-Design (Gold, eigene Komponenten, kein Fremd-Kit). Läuft isoliert als Sub-Agent.

05 · Bau API
/backend

APIs + Datenbank (PostgreSQL), Validierung mit Zod, echte Fehlermeldungen, Tests. KI nur über Langdock.

06 · Test
/qa

Testet gegen alle Kriterien und spielt Red-Team: Injection, DSGVO-Logging, „kein US-Provider". Urteil: reif oder nicht.

07 · Ship
/deploy

Dev → PR → IONOS. Mit nginx-Buffer-Fix, DNS-Check, Guard-Gate und Prod-Schutz. Login funktioniert von Anfang an.

08 · Ändern
/refine

Bestehende Spec überarbeiten, erweitern oder grundsätzlich in Frage stellen — sauber dokumentiert.

09 · Orientierung
/help

„Wo stehe ich, was ist der nächste Schritt?" — liest den Stand aus den Issue-Labels und führt weiter.

Was automatisch p2cx ist

Das Beste: Du musst an nichts davon denken. Es ist in den Skills eingebaut — jede App wird so geboren.

Born compliant
Guard-Score ab Tag 1

Alle 8 Standardmodule (Tracking, Error-Log, Health, Monitoring …) sind von Anfang an da. /deploy hat ein Guard-Gate: keine App geht unter Standard live.

DSGVO
Nur Langdock EU

Jeder KI-Aufruf läuft über Langdock (EU-gehostet). Kein OpenAI, kein Anthropic direkt, kein Fallback. QA prüft das sogar aktiv.

Design
p2cx-Look, deutsch

Gold/Grau/Grün/Rot, eigene Komponenten, SVG-Icons, keine Emojis, korrekte Umlaute. Kein fremdes UI-Kit.

Infrastruktur
PostgreSQL + Azure AD

Echte Datenbank auf IONOS, Login über NextAuth/Azure. Kein Prototyp-Modus — es entsteht eine richtige App.

Gelöst
Der Login-Bug ist Geschichte

Der nginx-Buffer für die großen Azure-AD-Tokens ist fest in /deploy. Plus DNS-Check vor SSL und automatische Port-Vergabe.

Nachvollziehbar
Alles in GitHub-Issues

Features, Specs, Test-Ergebnisse, Status — alles im Issue, mit Labels. Eine Quelle der Wahrheit, teamsichtbar.
